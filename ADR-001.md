# ADR-001: Serialised LLM Calls and Deferred OpenTelemetry

**Date:** 2026-04-02  
**Status:** Accepted  
**Deciders:** Project team  

---

## Context

The agent runs a local LLM via LM-Studio on a consumer GPU (typically 8–12 GB VRAM).
During initial development, two architectural pressures arose:

1. **VRAM exhaustion** – concurrent LLM requests saturate GPU memory and cause OOM
   crashes or severely degraded response times.
2. **Instrumentation scope** – integrating OpenTelemetry at generation time adds
   significant boilerplate and external dependencies that are not yet needed.

---

## Decisions

### 1. Serialise All LLM Calls with `asyncio.Semaphore(1)`

Every code path that calls LM-Studio must acquire `self._llm_semaphore` before
sending the HTTP request. The semaphore value is **1**, ensuring at most one
in-flight LLM call at any time.

**Rationale:**
- Eliminates OOM risk on GPUs with limited VRAM.
- Simplifies reasoning about concurrency; no need for request queuing logic.
- Acceptable trade-off: increased latency for parallel tasks vs. system stability.

**Consequences:**
- Throughput is limited to one LLM call at a time.
- Future work: allow semaphore value to be increased via `agent.yaml` once hardware
  is upgraded or quantisation reduces VRAM footprint.

---

### 2. Replace OpenTelemetry with `loguru` + SQLite Metrics

Rather than integrate OpenTelemetry traces and metrics (which requires a collector,
exporter, and additional dependencies), we use:

- **`loguru`** with JSON serialisation (`serialize=True`) for structured event logs.
- A minimal **SQLite `metrics` table** (`key TEXT PRIMARY KEY, value INTEGER`) for
  counter-based observability (e.g., `tasks_total`, `escalations_total`,
  `tool_executions_<name>`).

**Rationale:**
- Zero external infrastructure – logs and metrics are self-contained on disk.
- Dramatically reduces the number of required dependencies.
- SQLite metrics are readable by the FastAPI `/metrics` endpoint in
  Prometheus-compatible plain-text format.
- OpenTelemetry can be layered on top later (see Phase 8 notes) by wrapping the
  `json_log` and `metrics.inc` calls without changing call sites.

**Consequences:**
- No distributed tracing out of the box.
- Metrics are not scraped in real-time; they require the agent process to be running.
- Future work: add an OpenTelemetry exporter once the core loop is stable.
